Setting Up NAO

Before we begin programming, let's learn how to safely charge, set up, and handle the NAO robot. Taking care of these steps first ensures a smooth experience for you and a long life for the robot.

Charging NAO

1

Plug in the charging cable

When NAO is not moving, plug the charging cable into the robot's back. If the robot is charging, the charger's light will turn red. Always keep NAO charged between sessions.

Powering On

2

Place NAO in a stable position

Before turning on NAO, make sure it is in a stable seated position with its back upright and both feet flat on the floor. Do not place NAO on a table edge where it can fall and damage itself.

3

Press the chest button

Turn on NAO by pressing the chest button. NAO's lights will turn on and it can take up to several minutes to boot. NAO will make a sound when it has finished booting.

Connecting to NAO

4

Find NAO's IP address

Once NAO is fully booted, press the chest button briefly. NAO will speak its IP address aloud (e.g., 192.168.1.100). Write this down — you'll need it to connect from your code.

5

Ensure you're on the same network

Make sure your computer and NAO are connected to the same WiFi network or Ethernet. The default NAOqi port is 9559.

Safety First

Always handle NAO carefully. Support the robot's back when picking it up. Never let NAO walk on a table — it can easily fall off the edge. When in doubt, keep NAO in a seated position.

Before You Start Checklist

  • NAO is fully charged (or plugged in)
  • NAO is in a stable seated position on the floor
  • NAO has finished booting (you heard the startup sound)
  • You've noted the IP address
  • Your computer is on the same network as NAO
  • You have Python 2.7 installed (see next section)

Understanding ALProxy

ALProxy is the gateway to controlling NAO. It creates a connection (a "proxy") to a specific module running on the robot. The pattern is:

proxy = ALProxy("ModuleName", "robot_ip_address", 9559)

For example, to access text-to-speech:

# IMPORTANT: Run with Python 2.7
from naoqi import ALProxy

tts = ALProxy("ALTextToSpeech", "192.168.1.100", 9559)
tts.say("SDK connected!")

Some common NAOqi modules you'll use throughout this course:

  • ALMotion — Control NAO's movements and joints

  • ALSonar — Read distance sensors

Text-to-Speech Basics

NAO uses a text-to-speech (TTS) engine to convert text into spoken words. The ALTextToSpeech module handles this. In this section, you'll learn how to make NAO say anything you want.

How It Works

The process is straightforward:

  1. Your Python script connects to NAO's ALTextToSpeech module via ALProxy
  2. You call the say() method with a text string
  3. NAO's TTS engine converts the text to audio
  4. The audio plays through NAO's speakers

Your First "Hello World"

Here's the simplest NAO program — making it say "Hello World!":

# IMPORTANT: Run with Python 2.7
from naoqi import ALProxy

# Create a proxy to the text-to-speech module
# Replace the IP with your NAO's IP address
tts = ALProxy("ALTextToSpeech", "192.168.1.100", 9559)

# Make NAO speak!
tts.say("Hello World!")

Let's break this down:

  • from naoqi import ALProxy — imports the connection tool from the SDK
  • ALProxy("ALTextToSpeech", ip, port) — connects to NAO's speech module
  • tts.say("Hello World!") — sends text to be spoken. The say() method takes a string argument

Voice Parameters

NAO can speak faster, slower, higher, or lower by adjusting voice parameters. The two main parameters are speed and pitch.

Speed

The speed parameter controls how fast NAO speaks. The default value is 100.

  • Lower values (50–99): Slower speech
  • 100: Normal speed
  • Higher values (101–400): Faster speech
tts.setParameter("speed", 150)  # Speak 1.5x faster
tts.say("I am speaking quickly now!")

Pitch

The pitchShift parameter controls the pitch of NAO's voice. The default is 1.0.

  • Lower values (0.5–0.9): Deeper voice
  • 1.0: Normal pitch
  • Higher values (1.1–4.0): Higher voice
tts.setParameter("pitchShift", 1.5)  # Higher pitch
tts.say("I sound different now!")

Combining Parameters

You can change parameters between say() calls to create different voices:

# Deep, slow voice
tts.setParameter("speed", 70)
tts.setParameter("pitchShift", 0.7)
tts.say("I am speaking slowly and deeply.")

# Fast, high voice
tts.setParameter("speed", 200)
tts.setParameter("pitchShift", 1.8)
tts.say("Now I am fast and high pitched!")
Note

Parameter changes persist until you change them again. If you set the speed to 200, all following say() calls will use that speed until you call setParameter again.

Variables & Strings

Instead of hardcoding text directly into say(), you can use Python variables to build dynamic speech. This is much more powerful and reusable.

Storing Text in Variables

A variable stores a value that you can use later. A string is a sequence of characters enclosed in quotes:

robot_name = "Pablito"
greeting = "Hello! My name is " + robot_name + "."
tts.say(greeting)

String Concatenation

You can combine strings using the + operator:

first = "Hello"
second = "World"
message = first + " " + second + "!"
tts.say(message)  # NAO says: "Hello World!"

F-Strings (Formatted Strings)

F-strings are a cleaner way to embed variables inside text. Prefix the string with f and put variables inside {curly braces}:

name = "Pablito"
age = 10
message = f"Hi, I'm {name} and I'm {age} years old."
tts.say(message)

Building a Full Introduction

Combine what you've learned to have NAO give a complete introduction:

robot_name = "Pablito"
abilities = "walk, talk, and see"

tts.say(f"Hello everyone!")
tts.say(f"My name is {robot_name}.")
tts.say(f"I am a NAO humanoid robot.")
tts.say(f"I can {abilities}!")

The (x, y) Coordinate Plane

Before we can tell NAO where to walk, we need to understand how NAO sees the world around it. NAO uses a coordinate system centered on its own body to describe positions and directions.

NAO's Coordinate Frame

NAO's coordinate frame follows the standard robotics convention:

  • x-axis (forward/back) — Positive x is in front of NAO, negative x is behind
  • y-axis (left/right) — Positive y is to NAO's left, negative y is to NAO's right
  • z-axis (up/down) — Positive z is upward, negative z is downward

All distances are measured in meters. So x = 0.5 means "half a meter in front of NAO."

Understanding Theta (θ)

In addition to the x and y position, NAO needs to know its orientation — which direction it's facing. This is called theta (θ) and is measured in radians.

  • θ = 0 — NAO faces forward (along the +x axis)
  • θ > 0 — NAO turns left (counter-clockwise)
  • θ < 0 — NAO turns right (clockwise)
Radians vs Degrees

NAO uses radians, not degrees. A full circle is 2π radians (≈ 6.28). A quarter turn (90°) is π/2 (≈ 1.57). To convert: radians = degrees × π / 180.

Plotting Points

When we say "walk to point (1.0, 0.5)," we mean:

  • 1.0 meters forward (along the x-axis)
  • 0.5 meters to the left (along the y-axis)

The NAO robot always starts at the origin (0, 0) of its own coordinate system. Every time NAO moves, its coordinate system moves with it — this is called an egocentric (body-centered) reference frame.

Important

NAO's coordinate system is relative to the robot, not the room. If NAO turns 90° to the left, "forward" (+x) is now what used to be "left." This is different from a fixed, world coordinate system.

Making NAO Walk

Now that you understand NAO's coordinate system, let's make it walk! NAO's walking is controlled by the ALMotion module, and its posture (standing/sitting) by ALRobotPosture.

The ALMotion Module

The ALMotion module controls all of NAO's physical movements — walking, joint angles, stiffness, and more. The most important method for walking is:

motion.moveTo(x, y, theta)

This tells NAO to walk to a position relative to its current location:

  • x — distance forward (meters). Negative = backward.
  • y — distance to the left (meters). Negative = right.
  • theta — rotation angle (radians). Positive = turn left.

Standing Up and Sitting Down

Before NAO can walk, it must be standing. After walking, it's good practice to sit it back down:

from naoqi import ALProxy

posture = ALProxy("ALRobotPosture", "<robot_ip>", 9559)

posture.goToPosture("Stand", 0.5)   # Stand up (speed: 0.5)
# ... do things ...
posture.goToPosture("Sit", 0.5)     # Sit back down

The second argument is the speed (0.0 to 1.0). A value of 0.5 means half speed — safe and smooth.

Motor Stiffness

On a real robot, you need to turn on motor stiffness before NAO can move. Stiffness sends power to the motors:

motion.setStiffnesses("Body", 1.0)  # Turn on all motors
# ... walk and move ...
motion.setStiffnesses("Body", 0.0)  # Turn off (NAO goes limp)
Stiffness Safety

Always turn off stiffness when you're done. Leaving stiffness on for extended periods can overheat NAO's motors. In our browser simulator, stiffness is handled automatically.

Collision Detection

NAO has built-in collision avoidance for its arms and legs. You can enable or disable it:

motion.setExternalCollisionProtectionEnabled("Arms", True)
motion.setExternalCollisionProtectionEnabled("Legs", True)
Note

Collision detection helps prevent NAO from hitting objects while walking, but it's not perfect. Always supervise NAO when it walks, especially near table edges or stairs.

Polar Coordinates

So far we've described positions using Cartesian coordinates (x, y). But when NAO needs to walk to a point, it's often easier to think in polar coordinates — an angle and a distance.

From Cartesian to Polar

Given a target point (x, y), we can convert to polar coordinates (r, θ):

Distance (r):

r = sqrt(x2 + y2)

Angle (θ):

θ = atan2(y, x)

The Pythagorean Theorem

The distance formula comes directly from the Pythagorean theorem. If you draw a right triangle with legs x and y, the hypotenuse is the straight-line distance to the point:

import math

x = 1.0   # meters forward
y = 0.5   # meters left

distance = math.sqrt(x**2 + y**2)
# distance = sqrt(1.0 + 0.25) = sqrt(1.25) ≈ 1.118 meters

Why atan2 Instead of atan?

Python's math.atan2(y, x) is better than math.atan(y/x) because:

  • It handles all four quadrants correctly (returns angles from -π to +π)
  • It doesn't crash when x = 0 (division by zero)
  • It returns the correct sign for the angle
import math

x = 1.0
y = 0.5

theta = math.atan2(y, x)
# theta ≈ 0.4636 radians ≈ 26.57 degrees

The Walk-to-Point Strategy

To walk NAO to any (x, y) point, the strategy is:

  1. Calculate the angle to the target using atan2
  2. Turn to face the target: moveTo(0, 0, theta)
  3. Calculate the distance using sqrt
  4. Walk straight to the target: moveTo(distance, 0, 0)
import math

x = 1.0  # target x
y = 0.5  # target y

theta = math.atan2(y, x)             # angle to target
distance = math.sqrt(x**2 + y**2)    # distance to target

motion.moveTo(0, 0, theta)           # turn to face target
motion.moveTo(distance, 0, 0)        # walk straight there
Odometry

Odometry is how a robot tracks its own position by counting wheel rotations or step distances. NAO uses internal sensors to estimate how far it has walked. However, odometry accumulates small errors over time — NAO might think it walked 1.0m but actually walked 0.95m. This is called drift, and it's one of the biggest challenges in mobile robotics.

Walking with Python

Let's put everything together and write complete Python scripts that make NAO walk. We'll combine ALMotion, ALRobotPosture, and the math we've learned.

Script 1: Walk Forward

The simplest walking script — stand up, walk forward, sit down:

from naoqi import ALProxy

motion = ALProxy("ALMotion", "<robot_ip>", 9559)
posture = ALProxy("ALRobotPosture", "<robot_ip>", 9559)

posture.goToPosture("Stand", 0.5)
motion.moveTo(1.0, 0, 0)     # Walk 1 meter forward
posture.goToPosture("Sit", 0.5)

Script 2: Turn Then Walk

Turn 90° to the left, then walk forward:

import math
from naoqi import ALProxy

motion = ALProxy("ALMotion", "<robot_ip>", 9559)
posture = ALProxy("ALRobotPosture", "<robot_ip>", 9559)

posture.goToPosture("Stand", 0.5)
motion.moveTo(0, 0, math.pi / 2)   # Turn 90° left
motion.moveTo(0.5, 0, 0)           # Walk forward 0.5m
posture.goToPosture("Sit", 0.5)

Script 3: Walk to an (x, y) Point

Use trigonometry to walk to any point:

import math
from naoqi import ALProxy

motion = ALProxy("ALMotion", "<robot_ip>", 9559)
posture = ALProxy("ALRobotPosture", "<robot_ip>", 9559)

posture.goToPosture("Stand", 0.5)

# Target point
x = 1.0    # 1 meter forward
y = 0.5    # 0.5 meters left

# Calculate turn angle and distance
theta = math.atan2(y, x)
distance = math.sqrt(x**2 + y**2)

# Turn to face target, then walk straight
motion.moveTo(0, 0, theta)
motion.moveTo(distance, 0, 0)

posture.goToPosture("Sit", 0.5)

Putting It All Together

The general pattern for any NAO walking program is:

  1. Create ALMotion and ALRobotPosture proxies
  2. Stand up with goToPosture("Stand", speed)
  3. Walk using one or more moveTo(x, y, theta) calls
  4. Sit down with goToPosture("Sit", speed)
Tip

Each moveTo() call is relative to NAO's current position. If NAO turns 90° and then walks forward 1m, it walks in the direction it's now facing — not the original forward direction.

Intermediate

Q6. What is the difference between NAO's coordinate system and a room coordinate system?

NAO's coordinate system is egocentric (body-centered) — it moves with the robot. The x-axis always points where NAO is currently facing. A room coordinate system is fixed — directions don't change when the robot moves.

Q7. Why must you call goToPosture("Stand") before making NAO walk?

NAO must be standing to walk. The goToPosture method transitions NAO from sitting (or any other pose) to a stable standing position, ensuring the walking engine can function properly.

The distance (r = sqrt(x² + y²)) and the angle (θ = atan2(y, x)).

Q9. Why use math.atan2(y, x) instead of math.atan(y/x)?

atan2 handles all four quadrants correctly and doesn't crash when x = 0 (division by zero). It returns angles in the range (-π, π], giving the correct direction in all cases.

Q10. What is motor stiffness and why is it important?

Motor stiffness controls the power sent to NAO's motors. With stiffness at 0, motors are limp (no resistance). With stiffness at 1, motors are fully powered and can hold positions and move. It must be on for NAO to walk. Always turn it off when done to prevent overheating.

Q11. What is the exterior angle of a regular hexagon in radians?

2π / 6 = π/3 ≈ 1.047 radians (60°).

Advanced

Q12. What is odometry? What is its main limitation?

Odometry is the method of estimating a robot's position by tracking its movements (step counts, wheel rotations). Its main limitation is drift — small errors accumulate over time, causing the estimated position to diverge from the actual position. The longer NAO walks, the less accurate its position estimate becomes.

Q13. If NAO is at the origin and needs to reach point (0, -0.8), what moveTo calls would you use?

The point (0, -0.8) is directly to NAO's right. Strategy: turn 90° right, then walk forward: motion.moveTo(0, 0, -math.pi/2) then motion.moveTo(0.8, 0, 0). Alternatively: motion.moveTo(0, -0.8, 0) to sidestep directly (NAO can walk sideways).

Q14. Write the formula for the exterior angle of a regular n-gon. Explain why the total of all exterior angles is always 2π.

Exterior angle = 2π / n. The total is always 2π because walking around any closed polygon means making a complete 360° rotation. Since there are n equal turns: n × (2π/n) = 2π.

The two-step strategy (turn to face the target with moveTo(0, 0, theta), then walk straight with moveTo(distance, 0, 0)) is more predictable and accurate than diagonal walking. While moveTo(x, y, 0) does work, NAO's walking engine handles forward motion more accurately than combined forward+sideways motion. The two-step approach also makes the code easier to understand and debug.

Speech Recognition on NAO

Humans communicate through speech naturally, but for a robot this process is much more involved. NAO has four microphones built into its head that it uses to listen to sounds around it.

How NAO Hears

NAO's microphones are positioned around its head to capture sound from multiple directions:

  • Two front microphones (MicroFR, MicroFL) — Located on the front of the head
  • Two rear microphones (MicroRR, MicroRL) — Located on the back of the head

Unlike human ears that listen continuously, NAO must be programmed to listen for sounds at specific times. After hearing human speech, NAO performs speech recognition — using an algorithm to convert what it hears into words it can understand.

The Word Library

NAO requires a library of words that it expects to hear. For example, the library could contain "yes" and "no". When NAO hears something, it compares the sound against the words in its library and classifies it as one of those words — or as unrecognized.

Think of it like a phone menu

You've probably used automated phone services where you're given a list of options to say. NAO works similarly — it listens for specific words from a predefined list, not arbitrary speech.

The ALSpeechRecognition Module

In Python, we use the ALSpeechRecognition module to make NAO listen:

from naoqi import ALProxy

asr = ALProxy("ALSpeechRecognition", "<robot_ip>", 9559)
memory = ALProxy("ALMemory", "<robot_ip>", 9559)

# Set the language
asr.setLanguage("English")

# Set the word list (vocabulary)
vocabulary = ["hello", "goodbye", "yes", "no"]
asr.setVocabulary(vocabulary, False)

# Start listening
asr.subscribe("MyModule")

# ... wait for speech ...

# Check what was heard
result = memory.getData("WordRecognized")
# result is a list: [word, confidence, word, confidence, ...]

# Stop listening
asr.unsubscribe("MyModule")

Confidence Threshold

When NAO hears a word, it assigns a confidence score (0.0 to 1.0) indicating how sure it is about the match. A threshold controls how confident NAO must be before accepting a word:

  • Low threshold (0.1–0.3) — NAO accepts words easily but may misrecognize
  • High threshold (0.7–0.9) — NAO is very picky and may not recognize words you say
  • Recommended: 0.3–0.5 — A good balance between accuracy and responsiveness
Speech Recognition is Imperfect

NAO may mishear words, especially in noisy environments. The robot might hear "goodbye" when you said "hello." Adjusting the confidence threshold and speaking clearly helps improve accuracy.

Responding to Voice

Now that NAO can hear words, we need to program it to respond differently based on what it hears. This is where conditional statements come in — one of the most fundamental concepts in programming.

Conditional Statements (if / elif / else)

A conditional statement checks if a condition is true, and executes different code depending on the result:

# Basic if statement
if word == "hello":
    tts.say("Hi there! Nice to meet you!")

# if / else
if word == "yes":
    tts.say("Great! I'm glad you agree.")
else:
    tts.say("I didn't understand that.")

Handling Multiple Words with elif

When NAO's vocabulary has multiple words, use elif (short for "else if") to check each possibility:

if word == "nao":
    tts.say("Hello, I am a NAO humanoid robot.")
elif word == "r 2 d 2":
    tts.say("Hello, I am R 2 D 2.")
elif word == "c 3 p o":
    tts.say("Hello, I am C 3 P O, human cyborg relations.")
else:
    tts.say("I did not understand.")
How Conditionals Work

Python checks conditions from top to bottom. As soon as one condition is True, it runs that block and skips the rest. The else block runs only if none of the conditions above it were true.

String Concatenation in Responses

You can build dynamic responses using the word that was recognized:

# Using string concatenation
tts.say("Hello, I am " + word)

# Using f-strings
tts.say(f"You said {word}. That's interesting!")

Keyframe Motion

We move our own bodies fluidly, but robots are stereotyped to move in a clunky "robotic" fashion. Why? Some robots move in a "bang-bang" manner — suddenly moving a joint at maximum speed and then suddenly stopping. However, robots like NAO are capable of much more fluid motion.

What is a Keyframe?

A keyframe is a set of joint positions at a specific point in time. By defining a sequence of keyframes, you can describe complex motions for the robot.

  • Keyframe 1: NAO standing with arms at its sides
  • Keyframe 2: NAO standing with arms raised above its head

In keyframe motion, NAO's joints interpolate (transition smoothly) from one keyframe to the next. So for the example above, NAO would smoothly move its arms from its sides to above its head.

Interpolation

Interpolation means NAO calculates the intermediate positions between two keyframes and moves its joints smoothly through them. This is what makes the motion look natural rather than jerky.

Keyframes in Practice

Keyframe motion is used to design complex motions like:

  • Waving the robot's hands
  • Moving its feet for kicks (used in robot soccer!)
  • Dance routines
  • Gestures while speaking

Timeline and Frame Rate

Keyframes are placed on a timeline. Each number on the timeline represents a duration of 10 milliseconds:

  • Frame 10 = 100 ms (0.1 seconds)
  • Frame 15 = 150 ms (0.15 seconds)
  • Frame 100 = 1000 ms (1 second)

The spacing between keyframes controls the speed of motion — keyframes closer together produce faster movements.

Timing Matters

Don't place a keyframe at frame 1 (10 ms) — NAO would try to jerk to the position almost instantly, which could cause it to fall. Start with a keyframe around frame 10 (100 ms) so NAO transitions smoothly from its initial position.

Balancing NAO

As you may have learned from physics class, objects have a center of gravity. An object's center of gravity affects whether its pose is stable. For rigid bodies, the object is stable if the center of gravity lies within the base of the object.

Center of Gravity

For NAO, the same concept applies:

  • When standing on both feet, the base lies between both feet
  • When standing on one foot, the base is only that foot
  • NAO's center of gravity is approximately located in its torso

Because NAO can change its "shape" (moving arms, legs, head), the center of gravity shifts. You can create complex postures while keeping NAO balanced!

Staying Balanced

When adjusting NAO's pose, follow these rules:

  1. Check that NAO's weight is resting on both feet evenly
  2. Sometimes both feet touch the ground but all weight is on one foot — test by trying to move each foot
  3. Keep the center of gravity within the base of balance
  4. If NAO says its motors are hot, turn stiffness off for a few minutes to let it rest
Safety Tips

Always hold NAO when turning stiffness on or off. Be sure NAO doesn't hit itself with its hands during arm movements. You may need intermediate keyframes for complex motions (like moving hands from hips to behind the back).

Testing Balance

A simple test: while stiffness is on, try to move each foot. If a foot has weight on it, it will feel much heavier and harder to move than a foot with no weight. If all the weight is on one foot, NAO is less stable.

Joint Control with Python

Instead of manually creating keyframes in a visual editor, we can control NAO's joints directly with Python using the ALMotion module. This gives us precise, programmable control over every joint.

NAO's Joints

NAO has 25 joints (degrees of freedom). The head joints are the simplest to start with:

  • HeadYaw — Turns the head left and right (yaw rotation)
  • HeadPitch — Tilts the head up and down (pitch rotation)

The angleInterpolation Method

The angleInterpolation method smoothly moves joints to target angles over a specified time:

motion.angleInterpolation(names, angles, times, isAbsolute)
  • names — List of joint names (e.g., ["HeadYaw", "HeadPitch"])
  • angles — Target angles in radians (e.g., [0.0, 0.0])
  • times — Time to reach the angle in seconds (e.g., [[0.5], [0.5]])
  • isAbsolute — True for absolute angles, False for relative

Example: Nodding Yes

To make NAO nod "yes," we move the HeadPitch joint up and down:

from naoqi import ALProxy

motion = ALProxy("ALMotion", "<robot_ip>", 9559)

names = ["HeadYaw", "HeadPitch"]
times = [[0.5], [0.5]]

# Face straight ahead
motion.angleInterpolation(names, [0.0, 0.0], times, True)

# Nod three times
for i in range(3):
    motion.angleInterpolation(names, [0.0, 1.0], times, True)   # Look down
    motion.angleInterpolation(names, [0.0, -1.0], times, True)  # Look up

# Return to center
motion.angleInterpolation(names, [0.0, 0.0], times, True)

Example: Shaking No

To shake "no," we move the HeadYaw joint left and right:

from naoqi import ALProxy

Q4. Where must the center of gravity be for NAO to remain balanced?

The center of gravity must be within the base of balance. When standing on both feet, the base is the area between both feet. When on one foot, the base is that single foot.

A Timeline box. It provides a visual timeline where you can place keyframes and edit joint positions.

Q6. Why is it not suggested to set a keyframe at the first frame of the timeline?

Setting a keyframe at frame 1 (10 ms) would cause NAO to try to reach that position almost instantly, resulting in a violent jerk that could cause it to fall. Starting at frame 10+ (100+ ms) gives NAO time to transition smoothly.

Intermediate

Motions that move the center of gravity outside the base of balance — such as leaning too far forward, swinging arms too aggressively, or leg movements that shift weight incorrectly.

Q8. Explain the behavior of a Counter box (for loop) in Choregraphe/Python.

A Counter box / for loop repeats a sequence of actions a set number of times. It has an initial value, step value, and final value. In Python: for i in range(n) repeats n times, with i counting from 0 to n-1.

4 times. The counter goes: 2, 4, 6, 8 (stops when it would exceed 10). In Python: for i in range(2, 10, 2) gives values 2, 4, 6, 8.

HeadYaw turns the head left and right (rotation around the vertical axis). HeadPitch tilts the head up and down (rotation around the horizontal axis).

Finite State Machines

A finite state machine (FSM) is an abstraction commonly used in computer science and robotics. It consists of a finite number of states and transitions between them.

A Real-World Example

Imagine you're preparing for an exam over the weekend. Your situation can be described by two features:

  • How prepared you are (prepared or unprepared)
  • How much energy you have (rested or tired)

This gives us four states:

  1. Unprepared and Tired (starting state)
  2. Unprepared and Rested
  3. Prepared and Tired
  4. Prepared and Rested (goal state)

Actions and Transitions

Actions cause transitions between states:

  • Sleeping — Makes you rested (tired → rested)
  • Studying — Makes you prepared but tires you out (rested → tired, unprepared → prepared)
  • Playing — Makes you both tired and unprepared (any state → unprepared and tired)
Key FSM Rule

An FSM state must completely describe the situation without needing external information. Only one state is active at a time. Transitions move the FSM from one state to another.

FSM for NAO

We can use FSMs to design robot behaviors. For example, a bumper-controlled LED system:

  • Features: Ear LEDs (on/off), Eye color (A/B)
  • States: (ears on, eyes A), (ears on, eyes B), (ears off, eyes A), (ears off, eyes B)
  • Events: Left bumper press (toggles eye color), Right bumper press (toggles ears)

Implementing an FSM in Python

from naoqi import ALProxy

leds = ALProxy("ALLeds", "<robot_ip>", 9559)
tts = ALProxy("ALTextToSpeech", "<robot_ip>", 9559)

# Define states
state = "happy"  # Starting state

# Simulate a series of events
events = ["smile", "frown", "smile", "frown"]

for event in events:
    if state == "happy":
        if event == "frown":
            state = "sad"
            leds.fadeRGB("FaceLeds", 0x0000FF, 0.1)  # Blue eyes
            tts.say("Oh no, I am sad now.")
        else:
            tts.say("I am still happy!")
    elif state == "sad":
        if event == "smile":
            state = "happy"
            leds.fadeRGB("FaceLeds", 0x00FF00, 0.1)  # Green eyes
            tts.say("Yay, I am happy again!")
        else:
            tts.say("I am still sad.")

Events vs Actions

Actions are things performed by choice (studying, playing). Events are caused by something external (a bumper press, a voice command, a timer expiring). Both can trigger state transitions.

Sensors & Actuators

Beyond the external sensors we've discussed, NAO has internal sensors and actuators that are critical to its operation.

Actuators: NAO's Motors

Actuators on a robot refer to its joint motors. NAO has 21 different motors that can be controlled separately:

  • 2 motors for the head/neck
  • 2 for each shoulder (4 total)
  • 2 for each elbow (4 total)
  • 5 for the hips
  • 1 for each knee (2 total)
  • 2 for each ankle (4 total)

Encoders

Each motor is coupled with a sensor called an encoder. The encoder measures how far the motor has turned — this is the motor's angle of rotation.

For example, the encoder on the elbow joint tells NAO whether the arm is straight or bent. Using all 21 encoders, NAO is aware of the pose of its entire body.

Kinematic Chains

Using the angles from its encoders, NAO can calculate the position of any body part relative to another. This is called a kinematic chain. It uses trigonometry to compute positions.

For example, with a two-link arm (upper arm + forearm), if you know the length of each segment and the elbow angle, you can calculate exactly where the hand (end-effector) is:

import math

# Arm segment lengths (cm)
upper_arm = 50
forearm = 60

# Joint angle
shoulder_angle = math.radians(60)  # 60 degrees
elbow_angle = math.radians(30)     # 30 degrees

# Position of wrist (end-effector)
x = upper_arm * math.cos(shoulder_angle) + forearm * math.cos(shoulder_angle + elbow_angle)
y = upper_arm * math.sin(shoulder_angle) + forearm * math.sin(shoulder_angle + elbow_angle)

You can read sensor values using ALMemory:

You can also use ALMotion to read and set joint angles directly:

Multi-tasking

We do many things simultaneously in our day-to-day life. Chatting while walking, eating while watching television. In computer science, this process is known as multi-tasking — performing many tasks in parallel.

What is a task?

A task is something that can be performed independently. Multi-tasking is the process of performing many tasks at the same time.

CPUs and Threads

Within each computer (and robot), there is a CPU (Central Processing Unit) that handles the processing of tasks. A CPU core can only execute one sequence of instructions at a time.

One way to achieve multi-tasking is to use a multi-core CPU, so that multiple sequences of instructions can execute concurrently. But NAO's CPU has only a single core. So how does it multi-task?

The answer is threads. Programs are split into threads — sequences of code to be executed. The operating system switches which thread is currently running on the CPU thousands of times per second. This switching is so fast that it appears all threads are running simultaneously.

Thread Scheduling

The operating system rapidly switches between threads on a single-core CPU:

Time → [Thread 1][Thread 2][Thread 3][Thread 2][Thread 3][Thread 1][Thread 3][Thread 2][Thread 1]

Notice that each thread gets varying amounts of time, and they don't always run in the same order. The OS decides when and how to switch between threads.

Multi-tasking on NAO

NAO can perform multiple behaviors at the same time: moving its arms, shaking its head, speaking, and flashing LEDs — all concurrently. Each behavior runs as a separate thread.

In Python, we can use the threading module to run multiple functions in parallel. This lets us make NAO do several things at once, just like behavior layers in Choregraphe.

import threading
import time

def task_a():
    for i in range(3):
        print("Task A - step " + str(i))
        time.sleep(0.5)

def task_b():
    for i in range(3):
        print("Task B - step " + str(i))
        time.sleep(0.5)

# Run both tasks at the same time
thread1 = threading.Thread(target=task_a)
thread2 = threading.Thread(target=task_b)

thread1.start()
thread2.start()

# Wait for both to finish
thread1.join()
thread2.join()

print("Both tasks complete!")

Behavior Layers

In Choregraphe, Behavior Layers allow you to define multiple behaviors that run as separate threads simultaneously. Each layer controls a different part of NAO's body or a different action.

What are Behavior Layers?

Each layer defines a separate thread that runs on NAO. One layer could move the arms, another could shake the head, a third could control LEDs — all at the same time.

Why Use Behavior Layers?

Consider the robot dance. In the Macarena (from Module 4), the arms move first, then the legs, sequentially. But in many dances, the arms, legs, and head need to move simultaneously. Behavior layers make this possible.

Example: Arms + Head Dance

Imagine creating a dance where:

  • Layer 1 (Arms): Alternate raising left and right arms, looping forever
  • Layer 2 (Head): Bob head up and down, looping forever
  • Layer 3 (LEDs): Flash eye LEDs different colors
  • Layer 4 (Speech): NAO says "Nao" repeatedly

All four layers start playing at the same time. Each one controls a different aspect of the robot independently.

Timing and Synchronization

You can control when each layer starts. For example:

  • Head starts shaking at 0.3 seconds
  • Arms start moving at 0.5 seconds
  • Walking starts at 0.15 seconds
  • LEDs flash immediately

Each layer has its own timeline and can loop independently. The beauty of using behavior layers is that you can change the speed of one aspect of the dance while keeping the rest constant.

Translating to Python

In Python, we use threading.Thread to replicate behavior layers. Each "layer" becomes a function that runs in its own thread:

import threading
import time
from naoqi import ALProxy

motion = ALProxy("ALMotion", "<robot_ip>", 9559)
tts = ALProxy("ALTextToSpeech", "<robot_ip>", 9559)
leds = ALProxy("ALLeds", "<robot_ip>", 9559)

def arm_layer():
    """Layer 1: Move arms up and down."""
    for i in range(4):
        motion.angleInterpolation(
            ["RShoulderPitch", "LShoulderPitch"],
            [-1.0, -1.0], [1.0, 1.0], True)
        motion.angleInterpolation(
            ["RShoulderPitch", "LShoulderPitch"],
            [1.0, 1.0], [1.0, 1.0], True)

def head_layer():
    """Layer 2: Bob head up and down."""
    for i in range(6):
        motion.angleInterpolation(["HeadPitch"], [0.3], [0.5], True)
        motion.angleInterpolation(["HeadPitch"], [-0.3], [0.5], True)

def speech_layer():
    """Layer 3: Say words while dancing."""
    for i in range(3):
        tts.say("Nao!")
        time.sleep(1.0)

# Start all layers simultaneously
threads = [
    threading.Thread(target=arm_layer),
    threading.Thread(target=head_layer),
    threading.Thread(target=speech_layer),
]
for t in threads:
    t.start()
for t in threads:
    t.join()
Joint Conflicts

Be careful which joints you control in each layer. If two layers try to move the same joint at the same time, the motions will conflict and NAO may become unstable. Each layer should control a different set of joints.

Odometry & Walking in Circles

Now let's combine the robot dance with walking. We'll have NAO do part of the robot dance while walking in a circle using odometry to know when it has completed a full rotation.

What is Odometry?

Odometry

Odometry is the use of sensor data (from encoders in the motors) to estimate the robot's change in position over time. NAO tracks how far it has walked and how much it has turned.

NAO's ALMotion module provides getRobotPosition() which returns the robot's estimated (x, y, theta) position in global coordinates based on where it first started.

setWalkTargetVelocity

Instead of moveTo() which walks to a specific point, we can use setWalkTargetVelocity() to set NAO's walking speed continuously:

# Parameters: vx, vy, vtheta, frequency
# vx: forward speed (-1 to 1)
# vy: sideways speed (-1 to 1)
# vtheta: rotational speed (-1 to 1, positive = counter-clockwise)
# frequency: step frequency (0 to 1)

motion.setWalkTargetVelocity(0.5, 0.0, 0.25, 1.0)
# Walks forward at half speed while turning

By setting both forward speed (vx) and rotational speed (vtheta), NAO walks in an arc. Continuously walking in an arc traces out a circle!

The simplifyAngle Function

As NAO rotates, the total angle keeps increasing beyond 2π. We need a helper function to normalize angles to the range [-π, π]:

import math

def simplify_angle(theta):
    """Normalize angle to [-pi, pi] in O(1) time."""
    # Using modulo for constant-time normalization
    theta = theta % (2 * math.pi)
    if theta > math.pi:
        theta -= 2 * math.pi
    return theta
Why O(1) matters

A naive approach using a while loop to subtract 2π is O(θ) — it gets slower for large angles. The modulo approach above runs in constant time regardless of the input.

Walking in a Complete Circle

The algorithm:

  1. Record the starting angle using getRobotPosition()
  2. Start walking in an arc with setWalkTargetVelocity()
  3. In a loop, check odometry to see how much we've rotated
  4. When total rotation reaches 2π (a full circle), stop walking

Odometry Error

Odometry is never perfectly accurate. After walking a "circle," NAO won't end up exactly where it started. This is because:

  • Wheel/foot slip causes small errors that accumulate
  • Motor encoders have limited precision
  • The floor surface affects traction

The difference between NAO's actual final position and its starting position is called the odometry error.

from naoqi import ALProxy

motion = ALProxy("ALMotion", "<robot_ip>", 9559)
posture = ALProxy("ALRobotPosture", "<robot_ip>", 9559)

            ["RShoulderPitch", "LShoulderPitch"],

            ["RShoulderPitch", "LShoulderPitch"],

        motion.angleInterpolation(["HeadYaw"], [0.5], [0.6], True)

    threading.Thread(target=head_dance),
    threading.Thread(target=speak_dance),
]

    t.start()
for t in threads:
    t.join()

Exercise 3: Walking in a Circle with Dance

Make NAO walk in a circle while simultaneously performing the robot dance (head bobbing and LED flashing). Use setWalkTargetVelocity for the walk and threading for the parallel dance. Stop when a full circle (2π rotation) is complete.

You can't use arm/leg behavior layers while walking (the walking motion needs those joints). But you can use head and LED layers. Create a dancing flag that your head/LED threads check — set it to False when the circle is complete so they stop.

import math
import threading
from naoqi import ALProxy

motion = ALProxy("ALMotion", "<robot_ip>", 9559)
posture = ALProxy("ALRobotPosture", "<robot_ip>", 9559)
tts = ALProxy("ALTextToSpeech", "<robot_ip>", 9559)
leds = ALProxy("ALLeds", "<robot_ip>", 9559)

posture.goToPosture("Stand", 0.5)

dancing = True

def head_layer():
    while dancing:
        motion.angleInterpolation(["HeadYaw"], [0.4], [0.5], True)
        motion.angleInterpolation(["HeadYaw"], [-0.4], [0.5], True)

def led_layer():
    colors = [0xFF0000, 0x00FF00, 0x0000FF, 0xFFFF00]
    i = 0
    while dancing:
        leds.fadeRGB("FaceLeds", colors[i % len(colors)], 0.5)
        i += 1

threads = [
    threading.Thread(target=head_layer),
    threading.Thread(target=led_layer),
]
for t in threads:
    t.start()

tts.say("Walking in a circle while dancing!")

for step in range(8):
    motion.moveTo(0.2, 0, math.pi / 4)

dancing = False
for t in threads:
    t.join()

motion.moveTo(0, 0, 0)
leds.fadeRGB("FaceLeds", 0x000000, 0.3)
tts.say("Circle dance complete!")

Basic

1. What does CPU stand for?
Central Processing Unit — the hardware component in a computer (and robot) that handles all computational operations and processing of tasks.
2. Explain, at a high level, how a single processor performs multi-tasking.
The operating system splits programs into threads and rapidly switches between them (thousands of times per second). Each thread gets a small slice of processing time. This switching is so fast that it creates the illusion that all threads are running simultaneously, even though only one executes at any given instant on a single core.

Intermediate

3. Explain the use for behavior layers.
Behavior layers allow NAO to perform multiple actions simultaneously — such as moving arms, shaking head, speaking, and flashing LEDs all at the same time. Each layer runs as a separate thread controlling a different part of the robot, enabling complex multi-tasking behaviors like dances.
4. Why does standing up straighter make the robot more susceptible to falling?
Standing straighter raises NAO's center of gravity. A higher center of gravity means the robot has less stability — even a small push or imbalance can more easily tip it over. When NAO squats slightly, the lower center of gravity provides more stability.

Advanced

5. Why do we need to use the simplifyAngle method when computing odometry?
As NAO rotates, the raw angle from getRobotPosition() can grow beyond the range [-π, π]. The simplifyAngle function normalizes the angle back to this range so we can correctly compute how much rotation has occurred. Without it, we'd get incorrect angle differences that could cause the robot to walk too far or stop too early.
6. Explain the use of the setWalkTargetVelocity function.
setWalkTargetVelocity(vx, vy, vtheta, frequency) sets NAO's continuous walking velocity. Unlike moveTo() which walks to a specific point and stops, this function makes NAO keep walking at the specified speed until told to stop. Setting both vx and vtheta makes NAO walk in an arc (circle).
7. Why do we need to call time.sleep in the main loop?
time.sleep() puts the thread to sleep so the CPU can focus on other tasks (like actually executing the walk). Without it, the loop would execute as fast as possible, consuming all CPU time checking odometry and leaving no processing time for the walking behavior.
8. Why do we have to set the walk velocity back to zero at the end of the loop?
Because setWalkTargetVelocity sets a continuous velocity — NAO will keep walking forever until told to stop. Setting all velocities to zero explicitly tells NAO to stop walking. Without this, NAO would continue walking even after the function ends.
9. Using only division, multiplication, addition, subtraction, and integer truncation, write a function that finds the remainder of x / y (i.e., implement the modulus operation without using %). 
def modulus(x, y):
    return x - y * int(x / y)
This works because int(x / y) gives how many whole times y fits into x, and subtracting that product from x gives the remainder.
